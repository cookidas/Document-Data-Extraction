## 2. 문제 정의

### 현재 상황 (As-Is)

- 담당 주무관 1인이 하루 200~500건의 신청서를 수기로 검수
- 신청서마다 원본 이미지와 입력 내용을 번갈아 확인하는 **시선 분산** 반복
- 주민번호, 계좌번호 등 단순 오류도 수작업으로 발견해야 함
- 하루 종일 반복되는 단순 검증 작업 → **신체적·정신적 피로 누적**
- 처리 완료 데이터를 별도로 취합/정리해야 하는 **후처리 부담**

### 목표 상황 (To-Be)

- 신청서 일괄 업로드 → OCR 자동 추출 → **신뢰도 기반으로 의심 항목만 집중 검수**
- 마우스를 올리면 해당 영역의 원본 이미지가 바로 표시 → **시선 이동 최소화**
- 기본 형식 오류(자릿수, 형식) 자동 필터링 → **1차 휴먼 에러 시스템 차단**
- 검수 완료 데이터 즉시 DB 저장 및 CSV/Excel 일괄 다운로드

## 3. 타겟 사용자 분석

### 페르소나

- **이름**: 김주무관 (가상)
- **역할**: 화성시 동탄면 농정담당 주무관
- **일과**: 오전 8시 출근 → 신청서 접수 → 하루종일 분류·검수 → 데이터 입력
- **Pain Point**:
  - OCR/AI를 완전히 신뢰하지 못함 → 직접 원본과 대조하고 싶음
  - 화면과 서류를 번갈아 보는 것만으로도 눈이 피로함
  - 실수로 잘못된 데이터가 DB에 들어갈까봐 항상 긴장
  - 농번기 집중 기간에는 처리 건수가 폭증함

### 핵심 UX 원칙

1. **신뢰할 수 있어야 한다** - AI 결과를 맹목적으로 수용하지 않아도 됨
2. **빠르게 훑을 수 있어야 한다** - 한 화면에서 의심 항목만 확인
3. **실수가 DB에 들어가지 않아야 한다** - 체크 없이 저장 불가

## 4. 기술 스택 및 선정 근거

### 확정 스택

| 영역          | 기술                              | 선정 이유                                                          |
| ------------- | --------------------------------- | ------------------------------------------------------------------ |
| Frontend      | HTML / CSS / Vanilla JS           | 1일 개발, 프레임워크 셋업 시간 절약. 프로토타입에 충분             |
| OCR           | Naver CLOVA OCR (Template)        | 양식화된 문서에 특화, 필드별 Confidence Score 반환, 무료 티어 존재 |
| Backend       | Python Flask 또는 Node.js Express | 가볍고 빠른 API 서버. 파일 수신 + SQLite 연동에 적합               |
| DB            | SQLite                            | 서버 없이 파일 기반 동작, 1일 프로토타입에 최적                    |
| 엑셀 내보내기 | SheetJS (xlsx.js)                 | 클라이언트 사이드에서 Excel 생성, 별도 서버 처리 불필요            |

### 제외된 기술 및 사유

| 기술                      | 제외 사유                                                             |
| ------------------------- | --------------------------------------------------------------------- |
| OpenCV 3D 평탄화 전처리   | Python 전용, 브라우저 실행 시 성능 심각 저하. CLOVA가 내부적으로 처리 |
| React / Vue               | 1일 개발에서 셋업 오버헤드가 이득보다 큼                              |
| MySQL / PostgreSQL        | 프로토타입에 서버형 RDB는 과도한 설정 비용                            |
| AI 법률 검수 (Gemini/GPT) | **신청서 양식 미수령으로 연동 범위 불명확. 추후 옵션으로 보류**       |

## 5. 시스템 아키텍처

```
[사용자 브라우저]
     │
     ├─ 파일 업로드 (30장 batch)
     │
[Backend (Flask/Express)]
     │
     ├─→ [Naver CLOVA OCR API]
     │         │
     │         └─ JSON 응답 (필드값 + Confidence Score + Bounding Box)
     │
     ├─→ [SQLite DB]
     │         └─ 검수 완료 데이터 저장
     │
     └─→ [프론트엔드 반환]
               └─ 신뢰도별 UI 렌더링 + 검수 인터페이스
```

### OCR 연동 주의사항 (현실적 제약)

- CLOVA Template OCR은 **도메인 등록 + 신청서 양식 업로드 후 템플릿 구성** 필요
- 신청서 양식을 내일 수령 예정이므로, **개발 초반에는 Mock 데이터로 UI 먼저 구성**
- 양식 수령 후 템플릿 영역 지정 → OCR 연동 순으로 진행
- **Mock 모드 / Real 모드 스위치**를 코드에 플래그로 두어 빠른 전환 가능하게 구성

## 6. 기능 요구사항 명세서

### 6-1. 핵심 기능 (Must Have)

#### F-01. 신뢰도 기반 색상 하이라이트

- **목적**: AI를 불신하는 담당자도 안심하고 사용할 수 있도록 투명성 제공
- **동작**:
  - CLOVA Confidence Score < 80% → 입력창 테두리 **빨간색** + "필수 확인" 라벨
  - 80% ≤ Score < 90% → **노란색** + "확인 권장" 라벨
  - Score ≥ 90% → **초록색** + "자동 인식" 라벨
- **조건**: 빨간색·노란색 항목은 담당자가 **체크박스 클릭 후에만 저장 버튼 활성화**
- **근거**: 담당자가 의심 항목만 집중 확인 → 검수 시간 70% 단축 기대

#### F-02. 크롭 돋보기 (Bounding Box 툴팁)

- **목적**: 원본 이미지와 OCR 결과를 동일 화면에서 즉시 비교
- **동작**:
  - 입력창에 마우스 호버 시 → 해당 필드의 CLOVA Bounding Box 영역만 크롭
  - 원본 이미지를 Canvas로 잘라 **커서 옆에 툴팁 팝업**으로 표시
  - 툴팁 크기: 약 240×80px, 적절히 확대하여 가독성 확보
- **구현 방식**: HTML5 Canvas `drawImage()` + clip 영역 지정, 별도 라이브러리 불필요
- **근거**: 별도 창/모니터 전환 없이 즉시 대조 가능 → 시선 이동 및 신체 피로 최소화

#### F-03. 실시간 형식 유효성 검사

- **목적**: 1차 휴먼 에러를 사람이 아닌 시스템이 차단
- **검사 항목** (신청서 양식 수령 후 확정):
  - 주민등록번호: 13자리 숫자 + 앞 6자리 생년월일 유효성
  - 휴대전화: 010-XXXX-XXXX 형식
  - 계좌번호: 숫자만, 최소 10자리 이상
  - 농지 면적: 숫자(소수점 허용), 음수 불가
- **UX**: 필드 이탈(blur) 시 즉시 오류 메시지 표시, 수정 전 저장 불가

#### F-04. 일괄 업로드 및 배치 처리 대기열

- **목적**: 한 번에 30장 업로드 → 처리되는 동안 다른 업무 가능
- **동작**:
  - 파일 선택 또는 드래그앤드롭으로 최대 30장 업로드
  - 업로드 즉시 OCR 처리 큐(Queue)에 추가 → 순차 처리
  - 처리 현황 패널: 처리 중 / 검수 대기 / 완료 건수 실시간 표시
  - 처리 완료된 신청서는 검수 대기 목록에 자동 노출

#### F-05. CSV / Excel 일괄 다운로드

- **목적**: 검수 완료 데이터를 기존 업무 시스템(Excel 기반)에 바로 붙여넣기 가능하게
- **동작**:
  - 검수 완료 표시된 건만 선택적으로 또는 전체 일괄 다운로드
  - SheetJS를 이용해 클라이언트에서 직접 xlsx 생성 → 서버 부하 없음
  - 파일명: `직불금_검수완료_YYYYMMDD.xlsx`

### 6-2. 부가 기능 (Should Have)

#### F-06. 처리 현황 대시보드

- 오늘 업로드 건수 / 검수 완료 건수 / 오류 의심 건수를 상단 카드 3개로 표시
- 담당자가 오전에 올려두고 점심 후 돌아왔을 때 전체 현황을 한눈에 파악

#### F-07. 키보드 단축키 지원

- `Tab` : 다음 입력 필드로 이동
- `Space` : 현재 항목 검수 완료 체크박스 토글
- `Ctrl + Enter` : 현재 신청서 저장 후 다음 건으로 이동
- **근거**: 하루 200~500건 처리 시 마우스 클릭 수를 획기적으로 줄임

#### F-08. 간단한 로그인 화면

- 아이디/비밀번호 입력 → 세션 기반 유지 (prototype 수준: 하드코딩 계정)
- 포트폴리오/발표 시 "개인정보 접근 통제를 고려했다"는 어필 포인트

## 7. 비기능 요구사항

### 7-1. 보안

- **전송 구간 보안**: HTTPS 사용 권고 문구 표시 (프로토타입은 localhost, 실 배포 시 필수)
- **데이터 최소화**: 업로드된 원본 이미지는 OCR 처리 후 서버에 영구 저장하지 않음 (세션 내 임시 보관 후 삭제)
- **접근 제어**: 로그인 없이 메인 기능 접근 불가 (F-08)
- **민감정보 마스킹**: 검수 완료 목록 화면에서 주민번호 뒷자리는 `******`으로 표시
- **발표 어필 포인트**: "개인정보보호법 제24조를 고려하여 민감정보 최소 노출 설계"

### 7-2. 성능

- 30장 일괄 업로드 시 CLOVA API를 병렬이 아닌 순차 처리 (무료 티어 Rate Limit 준수)
- 개별 OCR 처리 완료 즉시 해당 건을 검수 대기 목록에 표시 (전체 완료 대기 불필요)

## 8. 화면 구성 (Page Flow)

```
[로그인 페이지]
      │
      ▼
[메인 대시보드]
  ├─ 오늘 처리 현황 카드 (업로드 / 대기 / 완료)
  ├─ 파일 업로드 영역 (드래그앤드롭 or 파일 선택)
  └─ 처리 대기 목록 (신청서별 상태 표시)
      │
      ▼ (건 클릭)
[검수 화면]
  ├─ 좌측: 원본 이미지 뷰어
  └─ 우측: OCR 추출 결과 입력 폼
       ├─ 필드별 색상 테두리 (빨강/노랑/초록)
       ├─ 호버 시 크롭 돋보기 툴팁
       ├─ 실시간 형식 유효성 검사
       ├─ 의심 항목 체크박스
       └─ [저장하기] 버튼 (모든 체크 완료 시 활성화)
      │
      ▼
[완료 목록 / 내보내기]
  ├─ 검수 완료 건 목록 (민감정보 마스킹)
  ├─ 개별 수정 버튼
  └─ [CSV 내보내기] / [Excel 내보내기] 버튼
```

## 10. 현실적 보완 의견

### 위험 요소 및 대응 방안

| 위험 요소                      | 가능성 | 대응 방안                                           |
| ------------------------------ | ------ | --------------------------------------------------- |
| 신청서 양식 수령 지연          | 중간   | Mock 데이터로 UI 100% 완성 후 OCR은 마지막에 붙이기 |
| CLOVA 도메인 설정 시간 소요    | 높음   | 오전에 병렬로 CLOVA 도메인 및 템플릿 세팅 시작      |

| 30장 동시 처리 시 UI 블로킹    | 중간   | 비동기(async/await) 처리 및 진행률 표시로 UX 보완   |

### 추가 제안 (발표 차별화 포인트)

- **"AI 보조, 사람이 결정"** 슬로건: AI가 대체하는 것이 아니라 보조한다는 철학 강조
- **처리 시간 비교 수치**: 기존 건당 2~3분 → 시스템 도입 후 의심 항목만 30초 목표
- **확장 가능성 언급**: 향후 타 시군구 직불금 양식으로 확대 가능한 템플릿 구조
